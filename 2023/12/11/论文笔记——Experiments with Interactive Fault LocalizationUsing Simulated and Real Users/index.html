摘要

Fault localization is considered a difficult and time consuming activity. However, tool support for automated fault localization is still limited because state-of-the-art algorithms often fail to provide efficient help to the user. They usually offer a ranked list of suspicious code elements, but the fault is not guaranteed to be found among the highest ranks. In Spectrum-Based Fault Localization (SBFL) – which uses code coverage information of test cases and their execution outcomes to calculate the ranks –, the developer has to investigate several locations before finding the faulty code element. Yet, all the knowledge she a priori has or acquires during this process is not reused by the SBFL tool. We propose an approach in which the developer interacts with the SBFL algorithm by giving feedback on the elements of the prioritized list. We exploit contextual knowledge of the user about the next item in the ranked list (e. g., a statement), with which larger code entities (e. g., a whole function) can be repositioned in their suspiciousness. First, we evaluated the approach using simulated users incorporating two types of imperfections, their knowledge and confidence levels. On SIR and Defects4J, results showed notable improvements in fault localization efficiency, even with strong user imperfections. We then empirically evaluated the effectiveness of the approach with real users, which also showed promising results.

故障定位被认为是一项困难且耗时的活动。然而，自动故障定位的工具支持仍然有限，因为最先进的算法通常不能为用户提供有效的帮助。它们通常提供一个排名列表，显示可疑的代码元素，但不能保证故障一定在最高排名中找到。在基于光谱的故障定位（SBFL）中，该方法使用测试用例的代码覆盖信息和它们的执行结果来计算排名。开发人员必须在找到有问题的代码元素之前调查多个位置。然而，在这个过程中，开发人员在此期间拥有或获取的所有知识都没有被SBFL工具重复利用。

我们提出了一种方法，其中开发人员与SBFL算法进行交互，提供对优先级列表中元素的反馈。我们利用用户对排名列表中下一个项目（例如，一个语句）的上下文知识，从而可以重新确定较大的代码实体（例如，整个函数）的可疑性。首先，我们使用模拟用户评估了这种方法，考虑了两种类型的不完善性，即他们的知识水平和信心水平。在SIR和Defects4J上，结果显示即使在用户不完美的情况下，故障定位的效率也有显著提高。然后，我们通过真实用户对这种方法的有效性进行了经验性评估，也取得了有希望的结果。

图1展示了我们方法的概念概述。该过程从基于像Tarantula这样的一些传统SBFL方法计算的初始排名开始。然后，元素从列表的开头开始向用户展示，并且iFL引擎等待用户的反馈。用户调查建议的元素并给出以下答案之一：

1）找到了故障，

2）元素没有故障，其上下文也没有故障，

3）元素没有故障，但故障在上下文中的某处，或者

4）不知道。

根据用户的反馈，iFL引擎执行以下操作。在情况（1）下，流程终止，而在情况（4）下，它将按照通常的方式继续下一个可疑元素（这意味着在最坏的情况下，当开发人员没有背景知识时，该方法会回退到纯SBFL方法）。在其余两种情况下，iFL引擎会对可疑性分数进行调整，重新计算排名，并在下一次迭代中向用户展示新列表中的下一个元素。

可以采用不同的策略来进行这些调整，例如对得分进行比例减少或增加，这些得分对上下文和系统的其他部分不同等等。目前，我们采用以下方法：在情况（2）下，整个上下文（即函数）得分为0，而在情况（3）下，除了上下文之外的所有内容都被降为0。

灵感：用大模型的话可以细化这些分数的比例。不用完全降为0，这样会更加准确。

EVALUATION GOALS

评价第一部分，对于模拟用户，与传统的非交互SBFL方法相比，iFL在需要检测的本地化有效性方面有多大的提高？

RQ1：iFL对来自SIR存储库中的人工故障有什么改进？

RQ2：iFL对缺陷4J存储库的真实故障有什么改进？

RQ3：iFL对用户的不完美有多敏感？

评价第二部分，考虑到真实用户的实际故障查找任务，与没有访问iFL的控制组相比，在开发环境中访问iFL实现的用户是否能够找到更多或更快的错误。

RQ4：用户比没有访问该方法的用户能发现更多的bug，这是真的吗？

RQ5：用户可以比没有访问该方法的用户更快地发现错误，这是真的吗？

RQ6：真实用户如何主观评价iFL方法及其在开发环境中的实现？

为了将iFL方法与传统的SBFL方法进行比较， 我们将计算这两种方法的费用指标，然后将它们与传统的SBFL相对改进进行比较。由于在每次方法迭代中都会在一步中决定一个代码块，所以我们将每次迭代都视为相当于计算费用的一个等级位置。然后将为每个缺陷计算改进的量，并生成适当的平均值。（？？？）

最近的用户研究报告指出，在放弃之前，开发人员通常只会调查本地化方法提供的推荐列表中的前5个或最多前10个元素[5]，[9]。因此，任何超出这些阈值的改进排名位置可能会变得不太有用，无论它们能够实现多少相对改进。

因此，我们定义了**”启用改进 enabling improvement“**的概念，即传统的FL算法将故障方法排在大于10（或5）的位置，但iFL方法在不到10（或5）步内达到了故障方法的改进。

将用3个案例来表达启用改进

• (10, ∞] → (5, 10] 基本的FL分数大于10，而iFL在5到10步内找到了故障。

• (10, ∞] → [1, 5] 基本的FL分数大于10，而iFL在5步内找到了故障。

• (5, 10] → [1, 5] 基本的FL分数在5到10之间，而iFL在5步内找到了故障。
